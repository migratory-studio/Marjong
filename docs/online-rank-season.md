# 通信対戦 ランク（段位・シーズン）仕様

> 位置づけ：通信対戦（テスト中）に「アカウント本人のオンライン段位」を載せる。育成（師弟モード）の
> 縦軸「宝（異能段位＝弟子ごと）」とは**別レイヤー**＝プレイヤー自身の腕の証。実装は段階的。
> 関連：`docs/online-multiplayer-p0.md` / メモリ online-rank-season-design。

## 0. 設計思想（核との整合）

このゲームの核は「失敗しないサクセス／孤独な麻雀を共闘に」。天鳳式の段位降格＝損失回避ストレスは
体温と逆行する。よって：

- **段位は絶対に下がらない**（永続ラチェット・季跨ぎでも不可侵）。
- **実力より「プレイ回数」が支配的**。RP は均すと必ず増える額にし、着順は登るテンポにだけ効かせる。
- 競技ランクは「数値を見せない」原則の**例外**（opt-in の別契約）＝段位・RP を明示してよい。
- 競技→愛着の橋渡し：昇段は「その一局を共にした相棒」が祝い、シーズン報酬は関係性レイヤーへ。

## 1. プレイ前提（必須ゲート）

通信対戦は「誰の記録か」が要るため、入口で2段ゲートを通す。

1. **アカウント連携（ログイン）必須** … 未ログインなら通信対戦に入れない。Google ログインを促す。
2. **ユーザーネーム必須** … ログイン済でも未設定なら、最初にユーザーネーム入力を要求してから入場。

保存先：ユーザーネーム＝既存の `profile.profile.displayName`（→ `profiles.display_name` / `misc`）。
新スキーマ不要。バリデーション：trim 後 1〜12 文字・空不可。一意性はテスト中は問わない（表示名）。
変更は将来アカウント画面から。導線（実装）：home「通信対戦」押下 → `getUser()` 判定 → 未ログインは
authPrompt、ログイン済かつ displayName 空はユーザーネーム入力モーダル → 通過で `online-screen`。

## 2. 段位テーブル（九蓮宝燈／宝士モチーフ・称号に数字を混ぜない）

9段位。蓮が開く → 雲を抜ける → 宝士に至る登り。`next` = その段位のバー満タン量（最高位は null）。

| 段位 | 称号 | 読み | 次段までRP |
|---|---|---|---|
| 1 | 萌芽 | ほうが | 200 |
| 2 | 蓮蕾 | れんらい | 300 |
| 3 | 開華 | かいか | 450 |
| 4 | 白蓮 | びゃくれん | 650 |
| 5 | 瑞蓮 | ずいれん | 900 |
| 6 | 金蓮 | きんれん | 1200 |
| 7 | 昇雲 | しょううん | 1600 |
| 8 | 凌霄 | りょうしょう | 2200 |
| 9 | 宝士 | ほうし | —（最高位） |

※キャラ固有称号（神算鬼謀＝詩玥／天衣無縫＝凌雲 等）は段位名に流用しない（混同回避）。
※称号名は `world.md` 同期時に最終照合。数値は調整前提。

## 3. RP・シーズン算出（実装＝`src/progression/onlineRank.js`・純ロジック）

段位状態 `state = { dan:1..9, tierRp, seasonId, seasonScore }`。保存は `profile.onlineRank`。

**1局の ΔRP（着順依存・調整前提）**

| 人数 | 1位 | 2位 | 3位 | 4位 |
|---|---|---|---|---|
| 4人 | +60 | +20 | 0 | −30 |
| 3人 | +50 | +10 | −30 | — |

**適用ルール（`applyMatchToRank`）**
- `tierRp = max(0, tierRp + Δ)` … **下限0＝降格しない**。
- 満タン（`tierRp >= next`）で **昇段**し `tierRp = 0` から再スタート（オーバーフローは持ち越さない＝各段位は新しい登り）。最大Δ(+60) < 最小閾値(200) なので1局で2段は飛ばない。
- 最高位（宝士）は昇段せず tierRp が貯まり続ける（フレックス表示・実害なし）。
- 均すと net プラス → **回数が支配的**。

**シーズン（クオーター制・案A）**
- `seasonId = "YYYY-Qn"`（`finishedAt` の UTC 月から Q1〜Q4）。
- シーズンが変わったら **`seasonScore` だけ 0 リセット**。**段位・tierRp は不可侵**。
- `seasonScore += max(0, Δ)` … 当季の活動量。**順位表は seasonScore で並べる＝回数が一番効く**。

**整合性**：`online_results`（対局ログ）を正として `computeRankFromResults` で再計算できる
（増分適用と一致することをテストで担保）。meta は高速読み出し用キャッシュ。

## 4. データ

- `online_results`（既存・作成済）… 対局ログ＝履歴の正・再計算ソース。`finished_at` で季に振り分け。
- アカウント共通 `profile.onlineRank`（→ `profiles.misc.onlineRank` jsonb）… 現在の段位状態。**スキーマ追加不要**。
- ユーザーネーム＝ `profile.profile.displayName`（→ `profiles.display_name`）。**スキーマ追加不要**。

## 5. 昇段演出（実装フェーズ）

昇段時、その一局を共にしたキャラの立ち絵＋昇段セリフ（`voiceLines` の event に `danUp` を1本追加＝
既存パターン）。師匠3名（詩玥・ビビ・ルイナ）は固有文言を直接執筆。最高位（宝士）到達は
**口調が崩れる特別報酬**に。他キャラは当面テンプレ。

## 6. 実装順（安全な刻み）

1. ✅ 純ロジック `onlineRank.js` ＋ テスト `test/onlinerank.mjs`（段位/RP/シーズン/降格なし/fold一致）。
2. ログイン＋ユーザーネーム ゲート（home「通信対戦」導線・authPrompt・ユーザーネーム入力モーダル）。
3. 対局終了時に段位状態を更新（`recordOnlineResult` 後に `applyMatchToRank` → profile 保存）＋結果画面に段位/RP/昇段表示。
4. 昇段演出＋キャラ別 `danUp` セリフ。
5. シーズン順位表（seasonScore ランキング）。

## 7. 未決・宿題

- **順位表のクロスユーザー読み取り**：`online_results` の RLS は現状「本人のみ select」。全体順位表には
  公開ビュー／集計テーブル／サーバ集計のいずれかが要る（段階4-5で設計）。表示名は順位表用に
  denormalize（`online_results` に `username` 列追加）も選択肢。
- ユーザーネームの一意性・変更UI・NGワードは後段。
